# SoloAvalon 协作规范

本文件适用于整个仓库。后续 AI 助手和维护者在本项目中协作时，应优先遵守这里的约定。

## 语言约定

- 默认使用中文回答用户、撰写说明、总结变更和记录待办。
- 新增或更新项目文档时默认使用中文。
- Git 提交信息默认使用中文。
- 代码标识符、API 路径、命令、第三方库名称和错误原文可以保留英文。
- 修改已有英文代码注释或用户界面文案时，优先保持所在文件的既有风格；新增较长说明性文字时优先使用中文。

## Git 提交信息

提交信息采用类似 Conventional Commits 的中文格式：

```text
<类型>(<范围>): <中文摘要>

<中文正文，可选>
```

要求：

- 标题行必须使用中文摘要，简洁描述本次变更，不以句号结尾。
- 标题行建议不超过 72 个字符。
- `<范围>` 可选，但涉及明确模块时建议填写。
- 正文可选；当变更原因、取舍、兼容性影响或验证方式无法从标题看出时，应补充正文。
- 一次提交只描述一组相关变更，避免把无关修改混在同一个提交中。

常用类型：

- `feat`: 新增功能或用户可见能力。
- `fix`: 修复缺陷。
- `docs`: 文档变更。
- `test`: 测试新增或调整。
- `refactor`: 不改变行为的代码重构。
- `style`: 格式、排版、命名等不影响行为的调整。
- `chore`: 构建、依赖、脚本、配置或仓库维护。
- `perf`: 性能优化。
- `ci`: CI/CD 配置变更。

建议范围：

- `backend`: 后端整体。
- `frontend`: 前端整体。
- `api`: API 路由、请求或响应模型。
- `game`: 阿瓦隆规则、领域模型和事件。
- `ai`: AI 玩家、上下文和策略。
- `llm`: 模型配置、provider 和 prompt 契约。
- `storage`: SQLite schema、仓储和事件存储。
- `docs`: 文档。
- `tests`: 测试。
- `scripts`: 启动脚本和开发脚本。
- `config`: 示例配置和配置加载。

示例：

```text
feat(frontend): 增加对局日志导出入口
fix(game): 修正任务失败票结算条件
docs: 补充本地模型配置说明
test(storage): 覆盖事件日志恢复流程
chore(scripts): 优化一键启动依赖检查
```

当提交包含破坏性变更时，在正文中明确说明影响和迁移方式：

```text
feat(api): 调整对局详情响应结构

BREAKING CHANGE: `players` 字段改为按座位号排序的数组。
前端和外部调用方需要同步更新读取逻辑。
```

## 开发与验证

- 修改后优先运行与变更相关的最小验证，再按风险扩大验证范围。
- 后端常用验证命令：

```powershell
.venv\Scripts\python.exe -m unittest discover -s tests -v
```

- 前端常用验证命令：

```powershell
cd frontend
npm run build
```

- 文档-only 变更可不运行完整测试，但应检查 Markdown 内容、路径和示例命令是否准确。

## 项目边界

- 后端是唯一规则裁判；前端不应自行推断隐藏真相或绕过后端规则。
- AI prompt 只能使用真人玩家合法可见的信息和过滤后的公开事件历史。
- 不要提交真实 API Key、本地模型供应商配置、SQLite 数据库、日志或虚拟环境文件。
- 真实模型配置应放在被 git 忽略的 `config/llm_profiles.json`，示例值放在 `config.example/`。
- 修改用户已有未提交内容时，先理解其意图并尽量保持，不要无故覆盖或回退。
