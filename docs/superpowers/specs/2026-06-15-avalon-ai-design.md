# 单人 AI 阿瓦隆游戏设计文档

日期：2026-06-15

## 1. 项目目标

制作一款本地单机 Web 应用，让一名真人玩家可以独自游玩《阿瓦隆》。除真人玩家外，其余玩家全部由大语言模型驱动。第一版优先实现强策略型 AI，让 AI 能围绕身份、投票、任务结果和发言进行推理、欺骗和博弈。

第一版采用标准 5 人局，真人玩家随机身份，AI 玩家使用 OpenAI-compatible 接口调用国产或兼容模型。每个 AI 玩家可以绑定不同的 API 配置，用于比较不同模型的策略水平和发言质量。

## 2. 第一版范围

### 2.1 必须实现

- 本地单机 Web 应用。
- Python FastAPI 后端。
- React + TypeScript 前端。
- SQLite 本地数据库。
- 标准 5 人阿瓦隆规则。
- 1 名真人玩家 + 4 名 AI 玩家。
- 真人玩家随机身份。
- 固定轮流发言。
- 强策略型 AI 决策。
- 阶段化提示词与上下文管理。
- OpenAI-compatible 模型调用。
- 默认共用一套模型 API 配置，并支持每个 AI 玩家单独覆盖绑定。
- 完整对局日志保存、查看、删除和导出。
- 架构预留 6-10 人局、扩展身份和湖中仙女机制。

### 2.2 暂不实现

- 多真人在线联机。
- 云端账号系统。
- 移动端原生应用。
- 沉浸式桌游动画和复杂美术。
- 自由聊天式插话。
- 完整调试台 UI。
- 首版可选择真人身份。

## 3. 技术方向

### 3.1 后端

后端使用 Python + FastAPI，负责游戏规则、状态流转、AI 决策、模型调用、日志持久化和配置管理。后端是唯一规则裁判，前端不直接判断胜负或修改游戏真相。

推荐模块划分：

```text
backend/
  app/
    main.py
    api/
    core/
    game/
    ai/
    prompting/
    llm/
    storage/
    schemas/
```

核心原则：

- 规则引擎必须确定性运行。
- AI 只能基于被允许的信息做决策。
- 完整真相只保存在服务端游戏状态和数据库中。
- 模型调用与规则引擎解耦。

### 3.2 前端

前端使用 React + TypeScript，推荐 Vite 作为开发工具。第一版采用实用型游戏桌面，不做重装饰，优先保证信息清晰、操作明确、对局节奏顺畅。

推荐页面：

- 开局配置页。
- 游戏桌面页。
- 模型配置管理页。
- 对局日志管理页。
- 对局复盘详情页。

### 3.3 数据库

第一版使用 SQLite。它适合本地单机运行，易于备份和迁移，足够支持对局日志和模型配置管理。未来若扩展云端部署，可以切换 PostgreSQL。

## 4. 游戏规则设计

### 4.1 MVP 身份配置

标准 5 人局：

- 梅林 Merlin
- 刺客 Assassin
- 忠臣 Loyal Servant x2
- 爪牙 Minion x1

真人玩家参与身份随机分配，不提供前端选身份入口。后端可保留 seed 参数用于调试和复现。

### 4.2 阵营与胜利条件

善方：

- 梅林
- 忠臣

恶方：

- 刺客
- 爪牙

善方胜利路径：

- 任务成功达到 3 次。
- 随后刺客刺杀梅林失败。

恶方胜利路径：

- 任务失败达到 3 次。
- 善方完成 3 次任务后，刺客成功刺杀梅林。

### 4.3 任务人数表

5 人局任务人数：

```text
第 1 轮任务：2 人
第 2 轮任务：3 人
第 3 轮任务：2 人
第 4 轮任务：3 人
第 5 轮任务：3 人
```

5 人局中，每次任务只需要 1 张失败票即可失败。

### 4.4 对局流程

每一轮任务按固定阶段执行：

```text
1. 队长阶段
   当前队长选择任务队伍。

2. 轮流发言阶段
   从队长开始，所有玩家依次发言。
   真人玩家轮到时输入发言。
   AI 玩家轮到时由策略层和 LLM 生成发言。

3. 投票阶段
   所有人对当前队伍投赞成或反对。

4. 投票结算阶段
   赞成票过半则队伍通过。
   未过半则计入一次组队失败，并由队长顺延重新组队。
   若连续 5 次组队投票失败，后续本轮进入强制组队：跳过投票环节，由当前队长选择队伍并直接通过。

5. 任务执行阶段
   队伍成员提交成功或失败。
   善方只能提交成功。
   恶方可以选择成功或失败。

6. 任务结算阶段
   记录任务成功或失败。
   若达到胜负条件则进入终局或刺杀阶段。

7. 刺杀阶段
   善方完成 3 次任务后，刺客选择一名玩家刺杀。
   若刺中梅林，恶方胜利；否则善方胜利。
```

## 5. 可扩展规则架构

虽然第一版只实现 5 人局，但规则层不能写死 5 人局。建议用配置化对象描述局面。

核心扩展点：

- `PlayerCount`：支持 5-10 人。
- `RoleSet`：描述本局启用身份。
- `RoleKnowledge`：描述每个身份开局能看到的信息。
- `QuestTable`：描述不同人数下每轮任务人数和失败票要求。
- `SpecialMechanic`：挂载湖中仙女等特殊机制。
- `WinCondition`：描述基础胜负和扩展身份影响。

后续可扩展身份：

- 派西维尔 Percival
- 莫甘娜 Morgana
- 莫德雷德 Mordred
- 奥伯伦 Oberon
- 湖中仙女 Lady of the Lake

湖中仙女应作为特殊机制插入阶段流，而不是硬编码进 5 人局流程。

## 6. AI 玩家设计

### 6.1 AI 目标

第一版 AI 以强策略为核心。AI 应尽量认真推理、控场、欺骗和反制，而不是主要追求角色扮演风格。

AI 应具备：

- 根据身份和可见信息形成初始判断。
- 根据投票、组队、任务结果和发言更新嫌疑。
- 善方尝试识别恶方，同时避免暴露梅林。
- 梅林尝试引导局势，但不能过度暴露。
- 恶方尝试误导善方，合理隐藏身份。
- 刺客在终局根据历史行为判断梅林。

### 6.2 AI 分层

AI 玩家建议拆为四层：

```text
AiPlayer
  -> ContextBuilder
  -> StrategyEngine
  -> PromptBuilder
  -> LLMProvider
```

`ContextBuilder` 负责为每个 AI 玩家生成合法私有视角，包括身份信息、公开事件、历史摘要和当前可选动作。

`StrategyEngine` 负责结构化判断，例如嫌疑评分、投票倾向、任务失败策略、刺杀候选人。

`PromptBuilder` 负责把私有视角、局面摘要、策略目标和输出格式组合成模型提示。

`LLMProvider` 负责调用具体 OpenAI-compatible 接口。

### 6.3 难度扩展

第一版默认使用强策略配置，但架构预留难度参数：

- 记忆长度。
- 推理深度。
- 发言详细程度。
- 欺骗强度。
- 错误率。
- 阵营协作程度。
- 风险偏好。
- 刺杀分析强度。

第一版可以不提供完整难度 UI，但后端模型中应预留 `difficulty_profile` 或等价配置。

### 6.4 提示词设计原则

提示词系统是 AI 玩家质量的核心，必须作为独立模块设计，而不是在业务逻辑中拼接零散字符串。

提示词设计原则：

- 每次模型调用都必须带有明确阶段目标，例如发言、投票、组队、任务行动或刺杀。
- 每次模型调用都必须基于当前 AI 玩家的合法私有视角。
- 模型不能接收该角色不应知道的隐藏身份、任务行动或系统真相。
- 模型输出必须是可解析的结构化结果，后端再根据规则校验并执行。
- 公开发言应像玩家发言，允许欺骗、隐藏意图和策略性模糊。
- 私有决策摘要应简短记录理由，不能要求或依赖模型输出完整隐藏思维链。
- 每个提示词模板应有版本号，方便复盘、调试和比较不同版本效果。

### 6.5 上下文分层

每次 AI 调用的上下文由 `ContextBuilder` 生成，按层组织：

```text
1. 固定系统规则
   阿瓦隆规则、当前启用身份、当前阶段规则、输出格式要求。

2. 角色私有信息
   当前 AI 的身份、阵营、合法已知角色信息和当前目标。

3. 当前公开局面
   轮次、队长、候选队伍、任务成功/失败数量、投票失败次数、已公开的历史结果。

4. 历史事件摘要
   最近若干条发言、投票、组队和任务结果，以及更早历史的压缩摘要。

5. 策略状态
   嫌疑评分、信任对象、风险判断、推荐行动和难度参数。

6. 当前可选动作
   本阶段允许的动作列表和动作参数限制。

7. 输出契约
   JSON schema、字段含义、禁止输出内容和失败重试要求。
```

`ContextBuilder` 是防泄漏边界。所有传给模型的信息都必须先经过它过滤，不能由调用方直接拼接完整游戏状态。

### 6.6 私有视角过滤

私有视角由 `RoleKnowledge` 统一生成。第一版 5 人局至少需要支持：

- 梅林知道恶方玩家，但在未来扩展莫德雷德时不能看到莫德雷德。
- 刺客和爪牙知道彼此是恶方。
- 忠臣不知道任何隐藏身份。
- 所有玩家都能看到公开发言、公开组队、公开投票结果和任务成败结果。
- 非任务队员不能看到单个任务行动。
- 复盘模式可以显示完整真相，但游戏进行中 API 和 prompt 都不能暴露。

前端状态过滤和 AI prompt 过滤应复用同一套视角规则，避免出现前端安全、AI 不安全的分裂实现。

### 6.7 阶段化提示词

不同阶段使用不同提示词模板和输出契约。

队长组队：

```json
{
  "team": ["player_id_1", "player_id_2"],
  "private_reason_summary": "选择这支队伍的简短理由",
  "public_message": "可选，队长组队时的公开解释"
}
```

轮流发言：

```json
{
  "public_message": "面向其他玩家的发言",
  "stance": "support_team | oppose_team | uncertain",
  "private_reason_summary": "本次发言策略的简短摘要"
}
```

组队投票：

```json
{
  "vote": "approve | reject",
  "private_reason_summary": "投票理由摘要",
  "public_reason": "可选，若界面需要展示简短表态"
}
```

任务行动：

```json
{
  "mission_action": "success | fail",
  "private_reason_summary": "任务行动理由摘要"
}
```

刺杀：

```json
{
  "target_player_id": "player_id",
  "private_reason_summary": "刺杀判断摘要",
  "candidate_ranking": ["player_id_1", "player_id_2"]
}
```

后端必须校验模型输出。若模型输出非法动作，例如善方提交失败任务、选择不存在的玩家、组队人数错误，应拒绝该输出并触发重试或 fallback。

### 6.8 历史压缩与记忆管理

完整对局日志保存在数据库中，但不能每次都把完整日志塞进模型上下文。AI 上下文应使用分层记忆：

- 近期窗口：保留最近若干条原始发言、投票、组队和任务结果。
- 回合摘要：每轮任务结束后生成结构化摘要。
- 玩家画像：记录每个玩家的嫌疑分、信任分、关键行为和发言倾向。
- 私有笔记：每个 AI 玩家拥有自己的私有判断，不同 AI 的记忆可以不同。
- 全局事实：由规则引擎维护，不由模型自由改写。

记忆更新应尽量结构化，例如：

```json
{
  "suspicions": {
    "player_id": 0.72
  },
  "trusted_players": ["player_id_1"],
  "key_observations": [
    "第 2 轮任务失败后，Player 3 仍强烈支持原队伍"
  ],
  "merlin_candidates": ["player_id_2", "player_id_4"]
}
```

模型可以参与生成摘要，但最终写入长期记忆前必须经过后端 schema 校验。对于关键事实，例如投票结果、队伍成员、任务结果，以规则引擎记录为准，不采信模型复述。

### 6.9 Token 与上下文预算

每个模型配置可以设置上下文预算。`ContextBuilder` 根据预算选择上下文内容：

- 优先保留当前阶段规则、合法动作和当前公开局面。
- 优先保留最近一轮的发言和投票。
- 较早历史使用摘要替代原文。
- 超出预算时压缩玩家画像和旧发言。
- 不因压缩丢失硬规则、身份私有信息和当前可选动作。

上下文预算不足时，系统应记录 `context_truncated` 标记，方便复盘时判断 AI 表现是否受上下文截断影响。

### 6.10 上下文缓存与增量策略

OpenAI-compatible Chat Completions 协议通常应按无状态接口设计。为了保持多厂商兼容，后端不能假设模型服务会记住上一轮对话，也不能只发送真正的增量内容。每次 AI 调用都必须包含足够让模型独立完成当前决策的上下文。

但上下文构建应尽量提高 prompt cache 命中率：

- 把稳定内容放在 prompt 前缀，例如通用游戏规则、身份规则、阶段规则、输出 JSON schema 和模板版本。
- 保持稳定前缀的文本、顺序和格式完全一致，避免在前缀中放时间戳、随机 ID、动态玩家状态或临时说明。
- 把动态内容放在后缀，例如当前私有视角、当前队伍、最近发言、嫌疑评分和可选动作。
- 对较早历史使用结构化摘要，避免每次重复发送完整原始日志。
- 每个 AI 玩家可复用同一套稳定规则前缀，但私有视角和私有记忆必须单独构建。
- 同一阶段的输出契约应稳定复用，减少格式变化造成的缓存失效。

推荐的上下文结构：

```text
cache-friendly stable prefix
  - Avalon base rules
  - enabled role rules
  - phase instructions
  - response schema
  - prompt template version

dynamic private suffix
  - player private view
  - current public state
  - recent event window
  - compressed memory
  - strategy hints
  - legal actions
```

如果某个模型服务支持显式 prompt cache、上下文缓存 ID 或会话续写能力，`LLMProvider` 可以在适配层中做可选优化。但核心游戏逻辑不能依赖这些厂商私有能力。没有缓存能力的兼容接口仍应能完整运行。

不要把“聊天窗口式增量发送”作为第一版基础机制。聊天产品看起来像只追加消息，但通常仍由应用或模型服务维护完整上下文、摘要或缓存。对本项目来说，后端数据库中的事件日志和 AI 记忆快照才是权威上下文来源，发送给模型的是由 `ContextBuilder` 每次重新生成的可裁剪视图。

### 6.11 Prompt 版本与调试

每次 AI 调用应记录：

- prompt 模板名称。
- prompt 模板版本。
- 上下文构建版本。
- 使用的模型配置。
- 上下文摘要。
- token 预算和是否发生截断。
- 是否使用缓存友好前缀。
- 稳定前缀版本或哈希。
- 模型输出解析结果。
- 后端校验结果。

第一版默认不在普通对局日志中展示完整 prompt 和原始模型响应，但可以在数据库中预留调试字段。导出日志时默认只导出摘要，后续可增加“包含 prompt 调试信息”的导出选项。

## 7. 模型配置设计

### 7.1 接口协议

只实现 OpenAI-compatible Chat Completions 协议。配置项包括：

- 配置名称。
- `base_url`
- `api_key`
- `model`
- `temperature`
- `max_tokens`
- `timeout`

适配目标包括国产或兼容 OpenAI API 的模型服务，例如 DeepSeek、通义千问兼容接口、智谱兼容接口，以及本地兼容网关。

### 7.2 玩家绑定

默认情况下，所有 AI 玩家共用同一个模型配置。开局配置可以为单个 AI 玩家设置覆盖配置：

```text
Default AI Profile -> LLM Profile A

AI Player 1 -> 使用默认配置
AI Player 2 -> 覆盖为 LLM Profile B
AI Player 3 -> 使用默认配置
AI Player 4 -> 覆盖为 LLM Profile C
```

这种设计让普通开局配置更简单，也保留同局比较不同模型、让强弱模型混合对局的能力。

### 7.3 安全要求

- API Key 不在普通游戏日志中明文保存。
- 前端不展示完整 API Key。
- 后端日志不输出完整 API Key。
- 导出对局日志时不包含 API Key。
- 如果保存模型原始请求和响应，必须过滤密钥字段。

## 8. 前端体验设计

### 8.1 游戏桌面

游戏桌面优先展示对局必要信息：

- 5 个座位。
- 玩家名称。
- 当前队长。
- 当前任务队伍。
- 真人玩家身份和可见信息。
- 任务成功/失败进度。
- 当前阶段。
- 组队投票历史。
- 任务结果历史。
- 轮流发言记录。

### 8.2 真人玩家操作

真人玩家根据阶段执行不同操作：

- 队长阶段：选择任务队伍。
- 发言阶段：输入发言。
- 投票阶段：赞成或反对队伍。
- 任务阶段：若在任务队伍中，提交任务行动。
- 刺杀阶段：若真人是刺客，选择刺杀目标。

不属于当前阶段的操作应禁用或隐藏。

### 8.3 配置管理

模型配置管理页应支持：

- 新增配置。
- 编辑配置。
- 删除配置。
- 测试配置是否可用。
- 隐藏 API Key 明文。

开局配置页应支持：

- 设置 4 个 AI 座位的名称。
- 选择默认 AI 模型配置。
- 可选地为单个 AI 座位覆盖模型配置。
- 启动新游戏。

## 9. 日志与复盘设计

### 9.1 日志目标

保存完整对局日志，并允许用户管理历史对局。日志既服务复盘，也服务 AI 调试。

### 9.2 必须记录的事件

- 游戏创建。
- 身份分配结果。
- 各玩家合法可见信息。
- 队长变化。
- 队长组队。
- 强制组队触发。
- 每位玩家发言。
- 每位玩家投票。
- 投票结果。
- 任务行动。
- 任务结果。
- AI 决策摘要。
- AI 上下文摘要。
- prompt 模板名称和版本。
- 模型输出解析与校验结果。
- 胜负结果。
- 刺杀选择和结果。

### 9.3 日志管理能力

第一版支持：

- 查看历史对局列表。
- 查看单局完整事件流。
- 删除历史对局。
- 导出单局日志为 JSON。

后续可扩展：

- 按模型配置筛选对局。
- 按胜负筛选对局。
- 查看 AI prompt 和原始模型响应。
- 对比不同模型表现。
- 展示嫌疑评分随时间变化。

## 12. 错误处理

### 12.1 模型调用失败

当 AI 模型调用失败时：

- 记录失败事件。
- 前端显示该 AI 响应失败。
- 后端可以使用一次重试。
- 重试失败后使用保守 fallback 决策。

fallback 决策示例：

- 发言：生成简短策略性表态。
- 投票：根据当前队伍中自身阵营和历史嫌疑进行规则决策。
- 任务：恶方根据失败进度和风险策略决定是否出失败票。

### 12.2 模型输出非法

当模型返回无法解析或违反规则的输出时：

- 记录原始输出摘要和校验失败原因。
- 使用同一上下文进行一次格式修复重试。
- 修复失败后使用 fallback 决策。
- 后端不得执行未通过规则校验的模型输出。

### 12.3 非法操作

后端拒绝所有非法阶段操作，例如：

- 非队长选队。
- 非当前发言者发言。
- 未在任务队伍中的玩家提交任务行动。
- 善方提交失败行动。
- 游戏结束后继续行动。

前端应尽量禁用非法操作，但最终校验必须由后端完成。

## 13. 测试策略

### 13.1 后端测试

必须重点测试规则引擎：

- 身份分配合法性。
- 每个阶段的状态流转。
- 5 人局任务人数。
- 投票通过和失败逻辑。
- 连续 5 次组队失败后触发强制组队通过，不触发恶方胜利。
- 任务成功和失败判定。
- 刺杀胜负判定。
- 私有视角过滤。
- 非法操作拒绝。

### 13.2 AI 测试

AI 测试分为确定性策略测试和集成测试：

- 给定局面时，策略层输出合理决策。
- Prompt 不包含 AI 不该知道的信息。
- ContextBuilder 对不同身份生成正确私有视角。
- 历史压缩不丢失当前阶段的硬规则和合法动作。
- 稳定 prompt 前缀在同一模板版本下保持一致。
- 动态私有视角只出现在后缀，不污染可缓存前缀。
- 阶段化 prompt 输出符合对应 JSON schema。
- 非法模型输出会触发重试或 fallback。
- AI 记忆快照只记录私有判断，不覆盖规则事实。
- LLMProvider 能调用兼容接口。
- 模型失败时 fallback 可用。

### 13.3 前端测试

前端重点测试：

- 不同阶段显示正确操作。
- 真人玩家身份信息显示正确。
- 发言、投票、组队、任务操作能提交。
- 历史日志可查看、删除和导出。

## 14. 实施顺序建议

后续进入实现阶段时，建议按以下顺序：

1. 建立后端项目骨架。
2. 实现 5 人局确定性规则引擎。
3. 实现 SQLite 存储和事件日志。
4. 实现模型配置管理。
5. 实现 ContextBuilder、PromptBuilder 和阶段化输出 schema。
6. 实现 LLMProvider 和 AI 策略接口。
7. 实现强策略 AI 的首版决策和记忆快照。
8. 实现前端游戏桌面。
9. 实现模型配置页面。
10. 实现日志管理和导出。
11. 做端到端本地运行验证。

## 15. 设计结论

第一版应采用方案 A：实用 MVP 架构。

核心目标不是立刻做成华丽桌游，而是先做出一款能认真游玩的单人阿瓦隆：规则可靠、AI 有策略、多模型可配置、日志完整可复盘。这个基础打稳后，可以自然扩展到 6-10 人局、更多身份、湖中仙女、调试模式和更沉浸的游戏界面。
