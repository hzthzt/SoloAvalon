# 随机座位与玩家匿名显示设计

## 背景

当前五人局创建时真人固定在 1 号座位，`Player.name` 会直接进入前端状态、公开事件复盘和 AI 上下文。新需求要求开局随机分配座位；座位确定后，所有对局公开称呼统一变为 `玩家{座位序号}`。玩家原名称只能用于前端玩家信息面板，且大模型全程不能看到这些原名称。

## 目标

- 开局时随机决定真人座位，AI 玩家填充其余座位。
- 对局中的玩家公开名称固定为 `玩家1` 到 `玩家5`，与座位序号一致。
- 真人昵称和 AI 原名称只在玩家信息面板显示，不进入 AI 上下文、公开事件、默认日志复盘或可导出的公开日志。
- 玩家信息面板继续显示真人玩家可见身份；其中真人玩家自己的真实角色保持可见。
- 现有基于 `player_1` 到 `player_5` 的操作、事件和持久化流程保持兼容。

## 方案选择

推荐方案是在后端领域模型中保留一个仅供 UI 使用的 `original_name` 字段，并让 `name` 始终表示对局公开匿名名。这样 AI 上下文和公开事件继续读取 `name` 时天然只会看到 `玩家N`，不会因为前端或 prompt 构建遗漏过滤而泄露原昵称。

备选方案是在前端维护原名称映射，后端完全不知道原名称。它能减少后端 schema 变更，但刷新、恢复历史对局和日志面板时会丢失原名称，不满足“玩家信息面板显示原名称”的稳定体验。

另一个备选方案是在每次构建 AI 上下文时过滤 `name`。它改动较少，但隐私边界分散在多个出口，后续新增事件或导出路径时更容易泄露。

## 数据模型

`Player.name` 表示公开匿名名，格式为 `玩家{seat_index + 1}`。新增 `Player.original_name: str | None` 表示开局前配置的原名称：真人来自创建请求中的真人昵称，AI 来自开局页的 AI 名称配置或默认 `AI N`。

SQLite `players` 表新增 `original_name` 列，用于恢复对局后仍能在玩家信息面板显示原名称。旧数据没有该列值时可回退为空。

## 创建流程

`create_five_player_game` 接收可选随机种子、真人原昵称和 AI 原名称列表。函数使用同一个随机源分别洗牌角色，并随机选择真人座位。生成玩家时：

- `id` 仍为座位绑定的 `player_{seat_index + 1}`。
- `seat_index` 仍为 0 基。
- `name` 固定为 `玩家{seat_index + 1}`。
- `original_name` 记录对应真人或 AI 的原名称。
- `is_human` 根据随机真人座位设置。

`GameService.create_game` 继续支持现有 `ai_names` 和模型配置；新增 `human_name` 输入。AI 模型覆盖仍按创建页的 AI 顺序输入，但保存时需要映射到随机落座后的非真人玩家，避免错误绑定到固定 `player_2` 到 `player_5`。

## API 与前端

创建对局请求新增可选 `human_name`。返回的 `players` 项新增 `original_name`，只供当前 UI 渲染玩家信息面板使用。前端开局页增加真人昵称输入，默认值可用 `真人玩家`。

玩家信息面板显示：

- 主标题：`玩家N`
- 辅助信息：原名称
- 身份：后端返回的 `visible_role`，其中真人自己的真实角色会显示，其他玩家仍按合法可见信息显示
- 类型：真人或 AI

组队勾选、刺杀目标下拉、信息流和日志复盘继续使用 `player.name`，因此只展示 `玩家N`。

## AI 隔离

AI 上下文构建不得包含 `original_name`，只保留 `id`、`seat_index`、`name` 和 `is_human` 等公开匿名信息。AI 的公开事件历史同样不包含原名称。新增测试需要直接检查 prompt 私有事件或上下文 JSON 中不出现真人昵称和 AI 原名称。

## 测试

- 规则层测试：同一 seed 下真人座位可不固定在 1 号位；所有公开名为 `玩家1` 到 `玩家5`；原名称保存在对应真人和 AI 玩家上。
- 服务层测试：创建返回包含 `original_name`，公开 `name` 为匿名名；AI prompt 不包含真人昵称和 AI 原名称。
- 持久化测试：`original_name` 写入并可从仓储恢复。
- 前端类型与构建测试：`PlayerView` 接收 `original_name`，玩家信息面板显示原名称，其他对局文本继续使用匿名名。

## 非目标

- 不改变玩家数量、角色池和任务配置。
- 不改变 `player_id` 的语义，也不为原昵称增加公开事件。
- 不支持手动指定真人座位；本次只实现随机座位。
