# 房间列表与 AI 用量复盘设计

## 背景

当前项目已经以 `games` 表保存每局游戏，并提供游戏列表、事件流、导出和 AI 决策记录。用户希望在主界面引入“房间”概念：房间列表展示每局游戏的状态，进入房间后查看完整信息流；已结束房间可以展开 AI 决策详情，并按玩家、按模型统计 token 消耗与缓存命中率。

本设计把“房间”定义为现有一局游戏的用户界面概念。后端继续以 `games` 作为权威存储，不新增独立 `rooms` 表。

## 目标

- 主界面提供房间列表，房间名使用 `game_id`。
- 新游戏 ID 改为 `游戏#Num`，不再使用时间戳。
- 房间状态展示为：正在游玩中、已结束、出现错误暂停。
- 进入房间后可以看到该局游戏的完整信息流。
- 已结束房间中，AI 决策事件可以展开查看提示词、AI 回答、token 消耗、缓存命中率等详情。
- 已结束房间中，按玩家和按模型展示总 token 消耗与平均缓存命中率。

## 非目标

- 不支持一个房间内多局游戏。
- 不估算 token 消耗或缓存命中率；模型接口未返回 usage 时显示暂无数据。
- 不把 AI 私有信息暴露给进行中的房间复盘入口，避免未结束对局泄露隐藏信息。

## 后端设计

### 房间与状态

复用 `games` 表作为房间列表数据源。`games.status` 保持面向存储的枚举值：

- `active`：正在游玩中。
- `complete`：已结束。
- `error_paused`：出现错误暂停。

API 可以继续返回存储值，由前端统一映射为中文状态。AI 决策失败时，服务层把当前游戏标记为 `error_paused`；后续手动重试或成功推进时恢复为 `active`。如果游戏已经产生胜负，则状态为 `complete`。

### Game ID 生成

删除时间戳 ID 生成方式，改为扫描已有 `游戏#Num` 格式 ID，取最大 `Num + 1` 生成新 ID。示例：

- 无历史房间：`游戏#1`
- 已有 `游戏#1`、`游戏#2`：`游戏#3`
- 历史中存在旧格式时间戳 ID：忽略旧格式，仅影响列表展示，不参与新编号。

生成逻辑放在 `GameService` 或 `GameRepository` 的明确方法中，由创建游戏流程调用。若插入时遇到唯一键冲突，则重新生成并重试一次，避免并发或旧数据造成冲突。

### AI usage 采集与存储

`LlmProvider.chat_completion` 当前只返回文本内容，需要改为返回包含内容与 usage 的结构。OpenAI 兼容响应中若存在以下字段，则采集：

- `usage.prompt_tokens`
- `usage.completion_tokens`
- `usage.total_tokens`
- `usage.prompt_tokens_details.cached_tokens`

如果字段缺失或不是数字，保存为 `null`。缓存命中率按 `cached_tokens / prompt_tokens` 计算，`prompt_tokens` 为 0 或缺失时为 `null`。

`ai_decisions` 表新增字段：

- `prompt_tokens integer`
- `completion_tokens integer`
- `total_tokens integer`
- `cached_tokens integer`
- `cache_hit_rate real`

现有数据库迁移通过 schema 初始化流程补齐缺失列。已有历史 AI 决策记录这些字段为 `null`。

### 决策详情与统计 API

事件流仍通过 `GET /api/games/{game_id}/events?include_private=true` 获取私有 payload，已结束房间才在前端展示 AI 私密详情。

新增或扩展一个房间详情数据契约，包含：

- 玩家列表与当前/最终游戏状态。
- 完整事件流。
- 每条 AI 决策的提示词、原始回答、解析结果、模型、profile、usage 和缓存命中率。
- 按玩家汇总：玩家 ID、玩家名、总 token、平均缓存命中率、决策次数。
- 按模型汇总：模型名、总 token、平均缓存命中率、决策次数。

统计只纳入有数值的 usage 字段。总 token 对缺失记录按 0 处理，同时保留决策次数；平均缓存命中率只对有缓存命中率的记录求平均，完全缺失时为 `null`。

## 前端设计

### 导航与页面结构

主界面从当前“对局 / 模型 / 日志”调整为更明确的房间体验：

- 对局：创建和进行当前游戏。
- 房间：房间列表和房间详情。
- 模型：模型配置。

实现上把现有“日志”页升级并重命名为“房间”，避免同时存在“日志”和“房间”两个含义重叠的入口。房间列表显示：

- 房间名：`游戏#Num`
- 状态：正在游玩中、已结束、出现错误暂停
- 人数、当前阶段、胜方、更新时间
- 操作：进入、导出、删除

### 房间详情

进入房间后右侧或详情区域展示：

- 完整复盘：沿用现有 `ReplayDetail`。
- 信息流：沿用现有事件流组件，显示该局所有事件。
- AI 决策详情：仅在已结束房间中，对 `ai_decision` 事件显示可展开详情。
- AI 用量统计：仅在已结束房间中显示玩家汇总和模型汇总。

进行中的房间仍可查看公开信息流，但隐藏 prompt、AI 回答、私有 payload 和统计详情。

### AI 决策展开内容

每条 AI 决策展开后显示：

- 决策玩家、阶段、决策类型、模型。
- token：prompt、completion、total、cached。
- 缓存命中率：百分比；缺失时显示“暂无数据”。
- prompt messages：按 role 分块展示。
- AI 原始回答。
- 解析结果与校验状态。

## 错误处理

- AI 决策异常会把房间状态标记为 `error_paused`，前端显示“出现错误暂停”并保留手动重试入口。
- usage 字段缺失不视为错误，展示“暂无数据”。
- 旧格式 game id 保持可访问、可删除、可导出，只是不参与新 ID 编号。
- 如果已结束房间没有 AI 决策记录，则统计区域显示空态。

## 测试计划

后端：

- 覆盖 `游戏#Num` 生成：空库、已有连续编号、混合旧格式 ID。
- 覆盖 AI 决策成功时 usage 持久化。
- 覆盖 usage 缺失时字段为 `null`。
- 覆盖 AI 决策失败时游戏状态进入 `error_paused`，成功推进后恢复 `active`。
- 覆盖按玩家、按模型汇总总 token 与平均缓存命中率。

前端：

- 覆盖房间列表显示中文状态和 `游戏#Num`。
- 覆盖进入房间后展示完整信息流。
- 覆盖已结束房间显示 AI 决策展开详情。
- 覆盖进行中房间不显示 prompt/回答等私密详情。
- 覆盖 usage 缺失时显示“暂无数据”。
